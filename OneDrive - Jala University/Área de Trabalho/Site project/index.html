ADR 0001 – Estilo arquitetural: Monolito modular com API REST

Status: Aprovado
Data: 2025-09-06
Autores: Arthur Oliveira, Ilmara Guimaraes, Sophia Goulart

Contexto

Projeto acadêmico/pequeno para registro de atividades diárias. Objetivo: entrega rápida, manutenção simples e deploy fácil.

Decisão

Aplicação monolítica modular com uma API REST que será consumida pelo front-end (react/flutter).

Alternativas Consideradas

Microservices — Pró: escala independente; Contra: complexidade, overhead operacional.
Single page app + Backend monolítico — Pró: entrega rápida ; Contra: limitação de escala granular.

Consequências

Simples deploy (um artefato backend + frontend estático).
Se precisar escalar, possivelmente separar serviços no futuro.

ADR 0002 – Framework do backend: Java + Spring Boot

Status: Aprovado
Data: 2025-09-06
Autores: Arthur Oliveira, Ilmara Guimaraes, Sophia Goulart

Contexto

Equipe com conhecimento em Java; objetivo de usar algo maduro para REST, segurança e JPA.

Decisão

Usar Java + Spring Boot para API REST.

Alternativas Consideradas

Node.js + Express — mais leve, rápida prototipagem; time prefere Java.
.NET Core — bom, mas não é linguagem principal do time.
Go — performático, porém curva de aprendizado maior para o time.

Consequências

Uso de Spring Security, Spring Data JPA e ecosystem maduro.
Necessidade de construir Dockerfile e pipeline CI.


ADR 0003 – Banco de dados: MySQL

Status: Aprovado
Data: 2025-09-06
Autores: Arthur Oliveira, Ilmara Guimaraes, Sophia Goulart

Contexto

Aplicação com relacionamentos simples (usuário ↔️ tarefas) e necessidade de consultas ACID.

Decisão

Usar MySQL como RDBMS.

Alternativas Consideradas

MongoDB — melhor para documentos, porém junções e relacionamentos de usuário/tarefa são triviais em SQL.
SQLite — bom para protótipo local, mas não para produção concorrente.

Consequências

Modelagem relacional, uso de índices em user_id e createdAt.
Necessidade de backups e migrações (usar Flyway ou Liquibase).
